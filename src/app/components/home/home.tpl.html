CSS Components

Have you ever started a project saying "Not this time... this time the CSS will be very well structured" and a week after it became a big mess? ...

... it has happened to me plenty of times

Thanks to CSS compilers like Sass or Less, we can arrange the code in different files and folders.

Although, it won't help us if we can't create a thoughtful structure for our code.

Design and Semantics: how to keep them together without messing each-other

In my experience these approaches lead us into issues:

  • Bind styles to tag selectors (chained or not)
    body section article h2 {...}
  • Using tags because how they look instead of what they mean
    <h1>Main Article title</h1> instead of <h2 class="main__title">Main Article title</h2>
  • Apply the same styles to different selectors over and over
    nav li a {cursor: pointer;} and after some lines aside li a {cursor: pointer;}

Design and Semantics: how to keep them together without messing each-other

How to feel you don't have to choose in between Design and Semantics, without writing yourself over and over:

  • Choose the markup for its meaning and not for how it looks by default
  • Create CSS components (if possible helped by some compiler) to standarize the team's code
  • Make those components with: variables, classes, mixins and some other ways to modularize code that might help you based on your experiences
  • Have fun creating them, is the best way to get more and more creative

Caching those values helps the dev team to not think over and over about them (cleaner way of working).

Also we could use a map to store them all into a single variable.

Standarize components, is not caching colors only. We can cache lot of different elements we use site-wide.

Helpers for generic tasks

For helper I mean a block of code which can be used multiple times (passing or not arguments). Some pros:

  • Keep all sort of modules in specific files and folders, so the whole team can access to them without re-doing what was already done
  • Creative tools can be used many times to enhance general and specific User Experience
  • Easier to adapt/modify some UI modularized before

Helpers for generic tasks

Classes Helpers
  • .transition: Add cool transitions to shifts
  • .clickable: Cursor pointer and avoid user selection. Very useful to make elements look like buttons despite they are not (overall web apps)
  • Headings: .main__title, .secondary__title, .third__title: Stop choosing headings by the way they look instead of what they mean (SEO meanining)
  • .text-ellipsis : Now a days most of projects are fluid/responsive and some elements just don't fit in the viewport. One of the best solutions is cropping texts without linebreaks (overall headings in mobile apps)
  • .antialiasing : With different devices (displays), browsers and fonts, the antialiasing technique becomes a must at the moment to preserve the project's look and feel. This Class is a mix of different techniques for different browsers
  • Retina display: .retina--visible and .retina--hidden: Display elements relying on DPI displays (to be included on HIDPI Mixin)

What is a mixin?

Mixins are Sass directives that allow us to create blocks of code passing or not arguments and also wrapping our code on itself.

One of the great advantages is that we can also set default values for the arguments in case we don't want to pass anything.

Another advantage is we can create a file for each mixin and troubleshoot easily.

Migrating from px into rem units:

When we speak about pixels (unit) we should know is not equal to a physical pixel in the display. It is just a unit and will vary depending the font in use.

Saying this, some devices can interpret "px" units different (for example Android).

To overcome this issue we should use relative measures.

"em" is one unit very often used on margins and paddings but for font-size is not that good since its size depends on its immediate parent.

For font practices the best unit is "rem" with is the same but relative to the root.

We can handle 1rem as 10px if we bind font-size: 62.5% into the "body" (which is the page's root).

pixel-rem mixin helps us to handle the "rem" units and also provide a "px" fallback for older browsers.
