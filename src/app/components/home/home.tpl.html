CSS Components

Have you ever started a project saying "Not this time... this time the CSS will be very well structured" and a week after it became a big mess? ...

... it has happened to me plenty of times

Thanks to CSS compilers like Sass or Less, we can arrange the code in different files and folders.

Although, it won't help us if we can't create a thoughtful structure for our code.

Design and Semantics: how to keep them together without messing each-other

In my experience these approaches lead us into issues:

  • Bind styles to tag selectors (chained or not)
    body section article h2 {...}
  • Using tags because how they look instead of what they mean
    <h1>Main Article title</h1> because is bigger than <h2>
  • Apply the same styles to different selectors over and over
    .navbar li {cursor: pointer;} and after some lines .sidebar li {cursor: pointer;}

Design and Semantics: how to keep them together without messing each-other

How to feel you don't have to choose in between Design and Semantics, without writing yourself over and over:

  • Choose the markup for its meaning and not for how it looks by default
  • Create CSS components (if possible helped by some compiler) to standarize the team's code
  • Make those components with: variables, classes, mixins and some other ways to modularize code that might help you based on your experiences
  • Have fun creating them, is the best way to get more and more creative

DRY Helpers

For helper I mean a block of code which can be used multiple times (passing or not arguments). Some pros:

  • Keep all sort of modules in specific files and folders so the whole team can access without redo what was already done
  • Creative tools can be used many times to enhance general and specific User Experience
  • It's easier to adapt/modify some UX modularized than reformat a lot of different selectors

DRY Helpers

Classes Helpers
  • .transition: Add cool transitions to shifts. This simple CSS3 helper will add category to any link, button or whatever you want while change their states.
  • .clickable: Cursor pointer and avoid user selection. Very useful to make elements look like real links and buttons (good practice on web apps components)
  • Headings: .main__title, .secondary__title, .third__title: Stop choosing headings by the way they look instead of what they mean. Pick your tags by their meaning and design apart.
  • .text-ellipsis : Now a days most of projects are fluid/responsive and some elements just don't fit in the viewport. One of the best solutions in block headings is cropping without linebreaks
  • .antialiasing : With different devices (displays), browsers and fonts, the antialiasing technique becomes a must at the moment to preserve the project's look and feel. This class nest browser-specific techniques with a text-shadow hack for fallback
  • Retina display: .retina--visible and .retina--hidden: Display elements relying on DPI displays (to be included on HIDPI Mixin)

Pixel-rem - Mixins Helpers

Migrating from px into rem units:

When we speak about pixels (unit) we should know is not equal to a physical pixel in the display. It's just a unit and will vary depending the font in use.

That said, some devices can interpret "px" units different (for example Android).

To overcome this issue we should use relative measures.

"em" is one unit very used on margins and paddings but for font-size is not that good since its size depends on its immediate parent.

For font practices the best unit is "rem" with is the same but relative to the root.

We can handle 1rem as 10px if we bind font-size: 62.5% into the "body" (which is the page's root).

pixel-rem mixin helps us to handle the "rem" units and also provide a "px" fallback for older browsers.

Images as <img> or background?

Let's figure when images should be actual image tags and when backgrounds.

First let's talk about design concepts: figure and background.

A figure is an image which is like a character in a play. Is content.

In the other hand, background is the scenario, decorates and ambient to strong the message. Is a secondary resource but just a little less important than characters.

Now that we understand this, we should recognize in design what images are intended to be actual pictures and those that are just decorative assets.

Pictures (characters) are actual image tags <img>.

Backgrounds and assets (scenario) are... background: ; (so no specific tag is required further from the proper semantic).
